Strict Review

この教材だけで十分か

初版を厳しめに見ると、方向性は正しい一方で「使いこなす」と言えるには、CLI、権限、文脈管理、検証、拡張機能の具体性が不足していました。

結論

この更新版では、現場エンジニアがClaude Codeを日常開発に投入するための基礎知識はかなり揃っています。ただし、API実装、料金試算、社内ポリシー、個別MCPサーバー設定は別教材で深掘りする前提です。

初版の弱点

  • CLIコマンドが代表例だけで、背景セッションや管理系が薄い。
  • slash commandの利用タイミングが見えにくい。
  • 権限モード、allow/ask/deny、設定スコープの説明が粗い。
  • 「実務ワークフロー」が抽象的で、依頼文と完了条件が不足している。
  • 最新モデルやプロンプト設計の前提を公式情報に寄せきれていない。

補強した判断基準

  • 小さい修正は直接実装、大きい変更はplan modeで調査と計画を分離する。
  • 作業依頼には必ず検証コマンド、期待結果、触ってよい範囲を書く。
  • チーム標準はProject設定、個人差分はLocal設定、強制ルールはManaged設定に置く。
  • 危険な権限はプロンプトでお願いせず、permission rulesやHooksで止める。

Chapter 1

Claude全体像

まず、Claudeを「チャットAI」ではなく、複数の利用形態を持つAIプラットフォームとして整理します。

3つの入口

  • Claude.ai: 会話、文章作成、調査、資料整理などの一般利用。
  • Claude API: 自社アプリや業務システムへの組み込み。
  • Claude Code: コードベースを読み、編集し、コマンドを実行する開発支援。

最初に押さえる概念

  • モデルは速度、知能、コストのバランスで選ぶ。
  • コンテキストはClaudeが一度に参照できる作業領域。
  • 最新のモデル名、価格、上限は変わるため公式ページで確認する。
領域 2026-06-13時点の押さえ方 現場での使い分け
Claude Fable 5 最も高い能力が必要な推論、長期タスク、高自律エージェント向けとして扱う。 難しい設計判断、長時間の調査、複数段階のエージェント作業。
Claude Opus 4.8 複雑な推論、agentic coding、高自律作業の有力候補。 設計、移行、重要PRレビュー、難しい障害調査。
Claude Sonnet 4.6 速度と知能のバランスを取りたい標準作業向け。 日常の実装、テスト追加、ドキュメント更新、差分レビュー。
Claude Haiku 4.5 高速応答が重要な軽量作業向け。 短い要約、分類、簡単な変換、軽量な補助タスク。
現場での見方

Claude Codeを導入する場合でも、Claude全般の限界、事実確認、モデル選択、プロンプト設計を理解しておくと、指示の精度とレビュー品質が上がります。モデル名、料金、提供状況は更新が速いため、教材更新時に必ず公式ページで再確認します。

Chapter 2

Claudeの基本操作

良い依頼は、役割、目的、前提、制約、出力形式が明確です。

依頼テンプレート

この型をベースにすれば、会話でもClaude Codeでもブレが減ります。

役割: あなたは社内研修教材を作るテックリードです。
目的: Claude CodeのCLI章を初心者にも分かる形で作成してください。
前提: 対象はWebエンジニア。Windows利用者もいます。
制約: 公式情報ベース。曖昧な機能名は断定しない。
出力: 章構成、説明本文、演習、注意点の順にMarkdownで。

要約

長い仕様書、議事録、ログを目的別に短くする。

比較

選択肢を評価軸つきで比較し、意思決定しやすくする。

レビュー

文章、設計、コード、テスト観点の抜けを見つける。

曖昧な依頼 現場で使える依頼 なぜ良いか
このコード直して ログイン失敗の原因を src/auth から調査し、再現テストを1つ追加してから最小差分で修正して。 範囲、期待成果、検証方法がある。
良い感じにレビューして この差分を correctness bug、security risk、missing test の3観点で重大度順にレビューして。スタイル指摘は最後に分けて。 レビュー観点と優先順位が明確。
ドキュメント作って 新規参加者が30分で起動できるREADMEを作って。必要コマンド、環境変数、よくある失敗、検証方法を含めて。 読者、利用場面、必須項目がある。

Chapter 3

プロンプト設計

プロンプトは「お願い文」ではなく、作業仕様です。Claudeに渡す情報の構造が成果物の品質を左右します。

実務で効く設計

  • 成功条件を先に書く
  • 入力データと指示を分ける
  • 出力形式を固定する
  • 例を1つ以上渡す
  • 不明点の扱いを決める
<task>
社内向けClaude Code研修のCLI章をレビューしてください。
</task>

<criteria>
- 初学者が迷わない
- Windows利用者への注意がある
- 公式情報に依存する箇所を明示する
</criteria>

<output>
重大な不足、改善案、次に作るべき演習の順で出力。
</output>

評価から始める

Anthropicの公式ガイドは、成功条件と評価方法を先に置くことを重視しています。プロンプト改善は、何を合格とするかが決まってから行います。

例を使う

Few-shot / multishotでは、実際の業務に近い例、例外パターン、望ましい出力を渡します。形式を安定させたい時ほど効果があります。

長文は構造化する

長い入力は上部に置き、最後に質問や作業指示を置きます。文書、制約、出力形式はXML風タグで分けると誤読が減ります。

Chapter 4

Claude Code入門

Claude Codeは、コードベースを調査し、編集し、検証コマンドを実行する開発エージェントです。

01

読む

ディレクトリ構造、関連ファイル、既存パターンを調べる。

02

計画する

変更範囲、リスク、検証方法を整理する。

03

編集する

小さく実装し、既存の設計に合わせる。

04

検証する

テスト、lint、ブラウザ確認、Git差分確認を行う。

Claude Codeの強さ

公式ドキュメントでは、Claude Codeはターミナル、IDE、Desktop、Webで利用でき、コードベースを読み、複数ファイルを編集し、コマンド実行やGit操作まで行う開発支援ツールとして説明されています。単発のチャットではなく、開発環境に入るAIとして扱います。

Chapter 5

Claude Code CLI完全ガイド

現場導入ではCLIが中核です。起動、継続、非対話実行、権限制御、トラブルシュートまでをまとめて扱います。

最初に覚えるコマンド

claude
claude "このプロジェクトの構造を説明して"
claude -p "このエラーログを要約して"
claude -c
claude --resume
claude doctor
claude --safe-mode

使い分け

  • 対話: claude で長めの作業を進める。
  • 初期指示付き: claude "..." で目的を渡して開始。
  • 非対話: claude -p でログ要約、差分レビュー、CI補助に使う。
  • 継続: claude -c または --resume で過去作業に戻る。
  • 切り分け: --safe-mode でCLAUDE.md、Hooks、MCP、Skillsなどの影響を外す。
分類 コマンド/操作 用途
起動 claude 対話セッションを開始する。
初期指示 claude "task" 目的を渡した状態で対話セッションを開始する。
ワンショット claude -p "..." 対話に入らず、標準出力へ結果を返す。スクリプト、CI、ログ要約向け。
入出力 --output-format json, stream-json -p と組み合わせ、機械処理しやすい形式で出力する。
継続 claude -c, --continue 現在ディレクトリの直近会話を続ける。
再開 claude --resume, -r セッションIDや名前、またはピッカーから過去セッションを再開する。
命名 --name, -n 再開しやすいようにセッション名を付ける。
診断 claude doctor, /doctor CLI、設定、接続、環境の問題を切り分ける。
対話内 /help, /clear, /compact, /context ヘルプ、会話リセット、文脈圧縮、コンテキスト使用状況確認。
シェル ! npm test 対話中にコマンドを実行して結果を共有する。
背景実行 Ctrl+B, --bg, claude agents 長時間コマンドやエージェント作業を背景で走らせ、一覧・接続・停止を行う。
権限 --tools, --permission-mode 利用できる操作や承認モードを制御する。
作業分離 --worktree, -w リポジトリ内の分離されたgit worktreeで作業する。
MCP --mcp-config, --strict-mcp-config MCPサーバーを指定し、必要なら他のMCP設定を無視する。
システム指示 --append-system-prompt, --system-prompt 通常はappendで追加する。置換はデフォルトのツール指針や安全指示も消えるため慎重に使う。
トラブル切り分け --safe-mode カスタマイズを無効化して、設定や拡張が不具合原因か確認する。

インストール

# Windows PowerShell
irm https://claude.ai/install.ps1 | iex

# macOS / Linux / WSL
curl -fsSL https://claude.ai/install.sh | bash

# WinGet
winget install Anthropic.ClaudeCode

非対話での実務例

git diff | claude -p "correctness bugを重大度順にレビューして"
cat test.log | claude -p "失敗原因と次の確認手順を要約して"
claude -p --output-format json "READMEの改善点をJSONで"

安全寄りの起動例

claude --permission-mode plan
claude --tools "Read,Bash"
claude --tools ""
claude --strict-mcp-config
claude --safe-mode
slash command 使うタイミング 現場での意味
/init, /memory 初回セットアップ、ルール整備 CLAUDE.md を作り、プロジェクト文脈を育てる。
/permissions 承認が多い、危険操作を制限したい allow / ask / deny ルールを管理する。
/plan 複数ファイル変更、未知の設計、影響範囲が大きい 編集前に調査と計画へ寄せ、誤実装を減らす。
/diff, /code-review, /review, /security-review commit前、PR前、重要変更後 差分確認、バグレビュー、PRレビュー、セキュリティ観点の確認。
/context, /compact, /clear 会話が長い、話題を切り替える 文脈の可視化、圧縮、新規タスクへの切り替え。
/agents, /fork, /batch, /tasks, /background 調査や大規模変更を並列化したい subagent、背景作業、worktree分割を使う。利用可否はプランや環境に依存する。
/resume, /branch, /rewind 過去作業に戻る、別案を試す、戻したい セッション継続、会話分岐、チェックポイントへの巻き戻し。
/doctor, /debug CLI、設定、MCP、Hooksが怪しい 不具合の診断とログ付き調査。
/schedule, /loop, /teleport, /remote-control 定期実行、監視、別端末やWebとの連携 クラウド/ローカルをまたぐ継続作業。利用条件は環境と契約で変わる。
Windowsでの注意

PowerShell、CMD、Git Bash、WSLでインストールコマンド、パス表記、シェルツールが変わります。公式Quickstartでは、Windows PowerShellには irm ... | iex、CMDには install.cmd、ネイティブWindowsではGit for Windows推奨という注意が示されています。チーム教材では、社内標準のターミナルを決めて手順を統一してください。

Chapter 6

実務ワークフロー

Claude Codeに丸投げするのではなく、調査、計画、実装、検証を短い単位で回します。

調査

関連ファイル、既存パターン、テスト構成を読ませる。

この不具合に関係しそうなファイルを探して、根拠つきで説明して

実装

変更範囲を限定し、既存設計に合わせる。

最小差分で修正して。不要なリファクタは避けて

検証

テスト、lint、ビルド、ブラウザ確認を行う。

変更後に実行すべき検証コマンドを選んで実行して

公開

Git差分を確認し、意図したファイルだけcommitする。

commit前に差分をレビューして、リスクとテスト結果をまとめて
タスク 推奨する進め方 完了条件
新しいコードベース理解 広く概要を聞き、次に認証、データモデル、テスト、ビルドなどに絞る。 主要ディレクトリ、実行コマンド、ドメイン用語、変更時の注意点を説明できる。
バグ修正 症状、再現手順、失敗ログ、期待結果を渡す。可能なら先に失敗テストを作る。 原因、修正差分、再発防止テスト、実行した検証コマンドが揃う。
リファクタ 挙動維持、変更範囲、既存パターン、後方互換性を明示する。 外部仕様が変わらず、テストが通り、差分の意図が説明できる。
テスト追加 未カバー箇所を探し、正常系、境界値、異常系を分けて追加する。 テスト名が仕様を表し、失敗時に原因が分かり、対象テストを実行済み。
PR作成 /diff と /code-review で確認し、要約、テスト結果、リスクを本文に入れる。 意図したファイルだけが入り、レビュー観点と検証結果が明示されている。
計画モードの使いどころ

公式のベストプラクティスは、未知の領域や複数ファイルにまたがる変更では、探索、計画、実装、commitを分けることを勧めています。一方で、タイポ修正や1行のログ追加など、差分を一文で説明できる作業は直接依頼して構いません。

Chapter 7

プロジェクト文脈の管理

Claude Codeの精度は、毎回の指示だけでなく、プロジェクトに置く文脈ファイルで大きく変わります。

CLAUDE.mdの最小テンプレート

チームの開発ルール、検証コマンド、禁止事項を短く書きます。

# Project Guide

## Commands
- Test: npm test
- Lint: npm run lint
- Build: npm run build

## Rules
- Follow existing component patterns.
- Do not change unrelated files.
- Run tests before summarizing completed work.

## Notes
- This project is used for internal training materials.
書きすぎない

CLAUDE.md は巨大な仕様書ではなく、毎回守ってほしい前提の圧縮版として使います。古い情報が残ると逆に精度が落ちます。

置き場所 向いている内容 注意
./CLAUDE.md または ./.claude/CLAUDE.md チーム共有のビルド、テスト、設計ルール、PR作法。 Git管理されるため個人情報やローカル事情は書かない。
./CLAUDE.local.md 個人のサンドボックスURL、手元だけのメモ。 .gitignore 対象にする。
~/.claude/CLAUDE.md 個人が全プロジェクトで使う作業好み。 プロジェクト固有ルールは置かない。
Managed policy 会社全体のセキュリティ、コンプライアンス、強制ルール。 IT/DevOps管理。ユーザー側で上書きできない前提にする。
Auto memory Claudeが作業中に学んだビルドコマンド、デバッグ知見、好み。 文脈であって強制設定ではない。禁止はHooksやpermission rulesで行う。

Chapter 8

応用機能

拡張機能は、現場課題に合わせて選びます。全部を最初から使う必要はありません。

Skills

特定作業の手順や知識を再利用可能な単位にする。PRレビュー、リリースノート、社内API移行など、毎回同じ手順がある作業に向く。

Subagents

レビュー、調査、テストなど役割ごとのエージェントに分ける。メイン会話の文脈を汚さず、専門タスクを並列に進めたい時に使う。

Hooks

編集前後やツール実行時にチェックや通知を差し込む。format、lint、secret検出、危険コマンド抑止など、機械的に守らせたいルールに使う。

MCP

外部ツールや社内データソースをClaude Codeから扱う。Jira、GitHub、Slack、Google Drive、社内DBなどをつなぐ時は権限範囲を先に決める。

GitHub Actions

PRやCIの文脈でClaudeを使う。CI失敗の要約、PRレビュー、issue triageなど、人間レビュー前の下処理に向く。

Agent SDK

独自のエージェント体験や自動化を構築する。CLIで足りない独自UI、権限制御、ワークフローを作る時の選択肢。

やりたいこと まず使うもの まだ足りない時
毎回同じ指示を守らせたい CLAUDE.md 内容が長いならSkillsへ分離。
危険操作を止めたい permission rules PreToolUse Hookで機械的に拒否。
レビューや調査を分担したい /agents, /fork /batch やworktreeで分割。
社内システムを読ませたい MCP Managed settingsで全社配布。
CI上で実行したい claude -p, GitHub Actions Agent SDKで専用フロー化。

Chapter 9

安全な導入

Claude Codeはファイル編集やコマンド実行ができるため、権限とレビューのルールが必須です。

人間が見るべきもの

  • 実行されるコマンド
  • 外部送信や認証が絡む操作
  • 削除、移動、権限変更
  • 機密情報や顧客情報を含むファイル

チームで決めるもの

  • 許可するツールとMCP
  • commit前の確認手順
  • AI生成コードのレビュー基準
  • ログ、監査、コストの扱い
制御対象 具体策 例
ツール利用 --tools でBuilt-in toolを絞る。MCPは別途denyやstrict設定で制御する。 claude --tools "Read,Bash"
承認モード --permission-mode で開始時の承認方針を決める。 default, acceptEdits, plan, auto, dontAsk, bypassPermissions
allow / ask / deny 設定ファイルで、許可するコマンド、都度確認する操作、禁止する操作を分ける。 Bash(npm run *) をallow、Bash(git push *) をdeny。
設定スコープ Managed、User、Project、Localを使い分ける。 全社セキュリティはManaged、チーム標準はProject、個人差分はLocal。
強制チェック プロンプトではなくHooksで止める。 secret検出、format、lint、危険な削除コマンドの拒否。

Chapter 10

ハンズオン演習

読むだけで終わらせず、現場で再現できる手順にします。

01

コードベース説明

任意のプロジェクトで claude を起動し、構造、主要ファイル、実行コマンド、注意点を説明させる。

02

ログ要約

claude -p にエラーログを渡し、原因候補、確認コマンド、担当者へ伝える要約を出す。

03

小さな修正

既存の不具合を最小差分で修正し、テストを実行する。

04

CLAUDE.md作成

プロジェクト固有のルール、検証コマンド、禁止事項を書く。

05

PRレビュー

/diff、/code-review、/security-review を使い、重大度順にリスクを整理する。

06

権限ルール

/permissions で、テスト実行は許可、pushや削除は確認または禁止にする。

07

計画モード

/plan で複数ファイル変更の設計だけ作らせ、人間が計画を編集してから実装へ進める。

08

背景作業

長いテストや調査を背景実行し、/tasks や claude agents で状態を確認する。

研修の最終課題

受講者がこの課題を完了できれば、Claude Codeの現場投入に必要な基礎は一通り身についたと判断できます。

1. 未知のリポジトリで claude を起動する。
2. /init で CLAUDE.md を作り、検証コマンドを追記する。
3. 小さな不具合を1つ選び、/plan で原因調査と修正計画を作る。
4. 実装後、対象テストを実行する。
5. /diff と /code-review で差分を確認する。
6. commit前に、変更ファイル、検証結果、残リスクを説明する。
