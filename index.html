Chapter 1

Claude全体像

まず、Claudeを「チャットAI」ではなく、複数の利用形態を持つAIプラットフォームとして整理します。

3つの入口

  • Claude.ai: 会話、文章作成、調査、資料整理などの一般利用。
  • Claude API: 自社アプリや業務システムへの組み込み。
  • Claude Code: コードベースを読み、編集し、コマンドを実行する開発支援。

最初に押さえる概念

  • モデルは速度、知能、コストのバランスで選ぶ。
  • コンテキストはClaudeが一度に参照できる作業領域。
  • 最新のモデル名、価格、上限は変わるため公式ページで確認する。
現場での見方

Claude Codeを導入する場合でも、Claude全般の限界、事実確認、プロンプト設計を理解しておくと、指示の精度とレビュー品質が上がります。

Chapter 2

Claudeの基本操作

良い依頼は、役割、目的、前提、制約、出力形式が明確です。

依頼テンプレート

この型をベースにすれば、会話でもClaude Codeでもブレが減ります。

役割: あなたは社内研修教材を作るテックリードです。
目的: Claude CodeのCLI章を初心者にも分かる形で作成してください。
前提: 対象はWebエンジニア。Windows利用者もいます。
制約: 公式情報ベース。曖昧な機能名は断定しない。
出力: 章構成、説明本文、演習、注意点の順にMarkdownで。

要約

長い仕様書、議事録、ログを目的別に短くする。

比較

選択肢を評価軸つきで比較し、意思決定しやすくする。

レビュー

文章、設計、コード、テスト観点の抜けを見つける。

Chapter 3

プロンプト設計

プロンプトは「お願い文」ではなく、作業仕様です。Claudeに渡す情報の構造が成果物の品質を左右します。

実務で効く設計

  • 成功条件を先に書く
  • 入力データと指示を分ける
  • 出力形式を固定する
  • 例を1つ以上渡す
  • 不明点の扱いを決める
<task>
社内向けClaude Code研修のCLI章をレビューしてください。
</task>

<criteria>
- 初学者が迷わない
- Windows利用者への注意がある
- 公式情報に依存する箇所を明示する
</criteria>

<output>
重大な不足、改善案、次に作るべき演習の順で出力。
</output>

Chapter 5

Claude Code CLI完全ガイド

現場導入ではCLIが中核です。起動、継続、非対話実行、権限制御、トラブルシュートまでをまとめて扱います。

最初に覚えるコマンド

claude
claude "このプロジェクトの構造を説明して"
claude -p "このエラーログを要約して"
claude -c
claude doctor

使い分け

  • 対話: `claude` で長めの作業を進める。
  • 初期指示付き: `claude "..."` で目的を渡して開始。
  • 非対話: `claude -p` でログ要約や差分レビュー。
  • 継続: `claude -c` で直近セッションを再開。
分類 コマンド/操作 用途
起動 claude 対話セッションを開始する。
ワンショット claude -p "..." スクリプトやパイプ処理で使う。
継続 claude -c 現在ディレクトリの直近会話を続ける。
再開 claude -r 過去セッションを選んで再開する。
診断 claude doctor CLIや設定の問題を切り分ける。
対話内 /help, /clear, /compact, /doctor セッション操作、文脈整理、診断を行う。
シェル ! npm test 対話中にコマンドを実行して結果を共有する。
権限 --tools, --permission-mode 利用できる操作や承認モードを制御する。
Windowsでの注意

PowerShell、Git Bash、WSLでパス表記やショートカットが変わります。チーム教材では、社内標準のターミナルを決めた上で手順を統一してください。

Chapter 6

実務ワークフロー

Claude Codeに丸投げするのではなく、調査、計画、実装、検証を短い単位で回します。

調査

関連ファイル、既存パターン、テスト構成を読ませる。

この不具合に関係しそうなファイルを探して、根拠つきで説明して

実装

変更範囲を限定し、既存設計に合わせる。

最小差分で修正して。不要なリファクタは避けて

検証

テスト、lint、ビルド、ブラウザ確認を行う。

変更後に実行すべき検証コマンドを選んで実行して

公開

Git差分を確認し、意図したファイルだけcommitする。

commit前に差分をレビューして、リスクとテスト結果をまとめて

Chapter 7

プロジェクト文脈の管理

Claude Codeの精度は、毎回の指示だけでなく、プロジェクトに置く文脈ファイルで大きく変わります。

`CLAUDE.md`の最小テンプレート

チームの開発ルール、検証コマンド、禁止事項を短く書きます。

# Project Guide

## Commands
- Test: npm test
- Lint: npm run lint
- Build: npm run build

## Rules
- Follow existing component patterns.
- Do not change unrelated files.
- Run tests before summarizing completed work.

## Notes
- This project is used for internal training materials.
書きすぎない

`CLAUDE.md` は巨大な仕様書ではなく、毎回守ってほしい前提の圧縮版として使います。古い情報が残ると逆に精度が落ちます。

Chapter 8

応用機能

拡張機能は、現場課題に合わせて選びます。全部を最初から使う必要はありません。

Skills

特定作業の手順や知識を再利用可能な単位にする。

Subagents

レビュー、調査、テストなど役割ごとのエージェントに分ける。

Hooks

編集前後やツール実行時にチェックや通知を差し込む。

MCP

外部ツールや社内データソースをClaude Codeから扱う。

GitHub Actions

PRやCIの文脈でClaudeを使う。

Agent SDK

独自のエージェント体験や自動化を構築する。

Chapter 9

安全な導入

Claude Codeはファイル編集やコマンド実行ができるため、権限とレビューのルールが必須です。

人間が見るべきもの

  • 実行されるコマンド
  • 外部送信や認証が絡む操作
  • 削除、移動、権限変更
  • 機密情報や顧客情報を含むファイル

チームで決めるもの

  • 許可するツールとMCP
  • commit前の確認手順
  • AI生成コードのレビュー基準
  • ログ、監査、コストの扱い

Chapter 10

ハンズオン演習

読むだけで終わらせず、現場で再現できる手順にします。

01

コードベース説明

任意のプロジェクトで `claude` を起動し、構造と主要ファイルを説明させる。

02

ログ要約

`claude -p` にエラーログを渡し、原因候補と次の確認手順を出す。

03

小さな修正

既存の不具合を最小差分で修正し、テストを実行する。

04

`CLAUDE.md`作成

プロジェクト固有のルール、検証コマンド、禁止事項を書く。

05

PRレビュー

差分をClaude Codeにレビューさせ、重大度順にリスクを整理する。
